コアバリューループ

Uchimiseのビジョン「発見と成長体験を届ける」を支える3つの循環構造。各ループが機能することで保存→調理転換率（KPI 30%）と30日継続率（KPI 50%）が達成される。

🔍 発見ループ
FeedContent / Creator ↓ 保存 Recipe（bookmark_status: saved） ↓ 調理 CookingRecord ↓ 学習 UserSkillStat → 次の発見提案
📅 計画ループ
Recipe（保存済み） ↓ 週次計画 MealPlan + MealPlanEntry ↓ 自動生成 ShoppingList + ShoppingItem ↓ 買い出し → 調理 CookingRecord
🌱 成長ループ
CookingRecord × CuisineGenre ↓ 蓄積 UserSkillStat（ジャンル習熟度） ↓ 可視化 ジャンル習得マップ ↓ 提案 「次のチャレンジ」→ 発見ループへ

提案エンティティ一覧

現行スキーマ（ARCHITECTURE.md）との差分を整理。🆕 = 新規追加、⚠️ = リファクタリング推奨。

エンティティ 種別 現状の問題 提案内容 バージョン
Creator 🆕 新規 recipes.creator_name がtext型のみ。クリエイターを個別に管理・参照できない 独立テーブル化。id / name / platform / channel_id / profile_image_url / genre_tags[]。Recipe.creator_id FK + UserFollowsCreator.creator_id FK で参照 MVP
Ingredient ⚠️ 正規化 recipes.ingredients が jsonb。型安全性なし。代替食材の ingredient_id 参照ができない recipe_ingredients テーブルへ正規化。{ recipe_id, name, amount, unit, order, is_optional } MVP
Substitution 🆕 新規 BRDに機能要件あり（代替食材提示）、DBスキーマなし substitutions { id, ingredient_id FK, alt_name, impact_note, source }。Ingredient 正規化が前提 MVP
Step ⚠️ 正規化検討 recipes.steps が jsonb。ステップ別タイマー実装・Step内の材料ハイライトが困難 recipe_steps テーブルへ。{ recipe_id, order, text, duration_minutes, tip }。優先度は Ingredient より低くてよい v1.0
ExtractionJob 🆕 新規 Phase1/2 のライフサイクルが recipes.phase / extraction_status に埋没。複数回リトライ・エラー履歴が追跡できない extraction_jobs { id, recipe_id FK, phase, status, error_code, attempts, started_at, completed_at }。レシピから分離してジョブ管理を明確化 MVP
MealPlanEntry 🆕 新規 MealPlan テーブルはあるが、何曜日・どの食事スロット（朝/昼/夜）にどのレシピを割り当てるかのエントリがない meal_plan_entries { id, meal_plan_id FK, recipe_id FK, date, meal_slot ENUM(breakfast/lunch/dinner) } MVP
ShoppingList 🆕 新規 v1.0 の「買い物リスト自動生成」機能に必須だがスキーマなし shopping_lists { id, user_id FK, meal_plan_id FK, generated_at, note } v1.0
ShoppingItem 🆕 新規 同上。複数レシピの同一食材の合算・カテゴリ分類・チェック状態の保持が必要 shopping_items { id, list_id FK, ingredient_name, amount, unit, category, is_checked, is_manually_added } v1.0
UserPreference 🆕 新規 AI献立提案・発見機能のパーソナライズに必須だが、ユーザーの好み・制約を保存するテーブルがない user_preferences { user_id FK, dietary_restrictions[], dislikes[], skill_level, max_cook_time_minutes, discovery_appetite } v1.0
CuisineGenre 🆕 新規 v1.5 の「ジャンル習得マップ」に必須。現状は recipes.tags[] に文字列として埋没しており集計不能 cuisine_genres マスタ { id, name, icon_name }。recipe_genres 中間テーブルで多対多。user_skill_stats で集計 v1.5
UserSkillStat 🆕 新規 ジャンル習得マップ・「次のチャレンジ」提案の集計元データが存在しない user_skill_stats { user_id FK, genre_id FK, cook_count, last_cooked_at }。CookingRecord 保存時に upsert で更新 v1.5
PantryItem 🆕 新規 v1.5 の食材在庫管理機能。「今日作れる」提案・賞味期限アラートに必須 pantry_items { id, user_id FK, name, amount, unit, category, expiry_date, added_at }。ShoppingList チェック時に自動追加する連携も考慮 v1.5
bookmark_status
（Recipe フィールド追加）		 ⚠️ フィールド追加 BRDに「お気に入り / 後で作りたい / 作り済み」のステータス管理とあるが recipes テーブルに該当フィールドなし recipes に bookmark_status ENUM('saved', 'want_to_cook', 'cooked') を追加。または recipe_bookmarks テーブルで分離 MVP
CookingSession 🆕 議論余地あり 調理モードUI（レシピ詳細）はあるが、「どのステップまで進んだか」「タイマー状態」を保存する仕組みがない。アプリ切り替え時に進捗が失われる cooking_sessions { id, cooking_record_id FK, current_step_order, started_at, timer_state_json }。必要性はUXリサーチで確認したい v1.0
MoodTag ⚠️ 分類整理 「気分で選ぶ」用タグ（あっさり/こってり/簡単/特別感）が通常の tags[] と混在。検索・フィルタリングで区別が難しい recipes.tags[] を roles に分けるか、recipes に mood_tags[] を追加して明示的に分離する MVP

自炊ドメインの見落とし・新視点

「料理・自炊」というドメイン全体で、現設計がカバーできていない視点。左ボーダーの色は重要度（🔴高 / 🟡中 / ⬜低）。

📝
作った後の評価・フィードバック
CookingRecord に note はあるが「また作る？」「難しかった？」の定量評価がない。保存→調理転換率 KPI の次のレイヤー「再調理率」の計測に必要。
候補: CookingRecord.rating / would_make_again / difficulty_actual
🔄
ユーザーカスタマイズレシピ
「代替食材を選択したらレシピが更新される」UXがあるが、元レシピと改変版の区別がない。ユーザーが手を加えたレシピをどう扱うか設計が必要。
候補: Recipe.is_customized / parent_recipe_id / RecipeVersion
💰
コスト・食費管理
ペルソナ「健康・コスパを重視」とあるが、レシピあたりのコスト概算・週次食費の見通しは設計されていない。買い物リストとの相性が良い。
候補: Ingredient.unit_price / Recipe.estimated_cost / ShoppingList.total_budget
🍂
旬・季節性
旬の食材を使ったレシピの優先表示は、「発見」体験を豊かにする要素。現状タグに依存しているが、時期に応じた自動推薦に発展できる。
候補: Ingredient.peak_seasons[] / Recipe.season_tags[]
🍳
調理器具・設備制約
「オーブンなし」「電子レンジのみ」などの設備制約はレシピの実現可能性に直結。UserPreference に含めるか専用の設計が必要。
候補: UserPreference.available_equipment[] / Recipe.required_equipment[]
🧑‍🤝‍🧑
人数・スケール変換
一人暮らしが基本だが、「たまに友人を呼ぶ」シチュエーションがある。レシピの serving_size に対して人数指定で材料量を自動スケールする設計が必要。
候補: Recipe.serving_size（追加）/ ShoppingItem での乗算処理
📊
栄養バランス
現在は意図的にスコープ外（PFCバランス詳細計算）だが、「ヘルシー」タグの精度向上には栄養メタデータが有効。将来の拡張余地として記録。
スコープ外（v2.0以降 or 外部API連携）
📅
作り置き・保存期間
「作り置き」タグは存在するが、作ったものを冷蔵・冷凍で何日保存できるかのデータがない。週次献立計画での作り置き活用提案に必要。
候補: Recipe.storage_days / Recipe.is_meal_prep_friendly
