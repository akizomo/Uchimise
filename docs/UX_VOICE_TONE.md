# Uchimise — Voice & Tone ガイド

> ブランドガイドライン v2.1 §4 の実装仕様。
> UIコピーを書く人・AIツールが迷ったときに戻る場所。
> 最終更新: 2026年3月

---

## 1. ブランドボイスの核心

### アナロジー：行きつけの食料品店の、物知りな店主

Uchimiseが話すとき、その言葉は**「AIが生成した提案」ではなく「店主がそっと差し出した一言」**として届く。

| 店主がすること | 店主がしないこと |
|---|---|
| 今日の仕入れをさりげなく教える | 「AIが最適化しました」と説明する |
| 「〜いかがですか」と一言添える | 「〜してください」と命令する |
| 常連の前回を覚えている | 連続記録を称える |
| 決定をユーザーに委ねる | 押しつける |
| 静かに喜びを共有する | 「すごい！」と叫ぶ |

---

## 2. ボイスの4原則

### 原則1｜丁寧語ベース、でも堅くない

- ✅ `〜しませんか` `〜いかがですか` `〜してみてください`
- ❌ `〜してください`（命令）
- ❌ `〜だよ` `〜だね`（タメ語）
- ❌ `〜でございます`（過剰な敬語）

### 原則2｜感嘆符（！）は使わない

感嘆符は押しつけがましさと義務感を生む。静かな自信と温かさで伝える。

- ✅ `できあがりました。`
- ❌ `できあがりました！`
- ✅ `レシピを棚においておきました。`
- ❌ `保存しました！`

### 原則3｜機能説明をしない

ユーザーが欲しいのは機能の説明ではなく、次に何をするかの気づき。

- ✅ `今週はタイカレーを試してみませんか。`
- ❌ `AIがあなたの保存履歴を分析して献立を提案します。`
- ✅ `材料と手順が整理されました。`
- ❌ `Phase 2 の処理が完了しました。`

### 原則4｜一言添えて、決めるのはあなた

提案は一言だけ。選ぶのはユーザー。理由を長々と説明しない。

- ✅ `大葉は最後に加えると、香りが立ちますよ。`
- ❌ `大葉はポリフェノールが豊富で加熱すると香りが揮発するため、最後に加えることをお勧めします。`

## 3. トーンダイヤル

状況によってトーンの温度を調整する。ボイス（キャラクター）は変えない。

```
冷静 ←─────────────────────────────→ 温かい

エラー    処理中    中立    提案    完了    発見
  |          |        |       |       |       |
 淡々と    静かに   端的に  一言で  静かに  嬉しそうに
 事実を   状況を   伝える  添える  喜ぶ    でも静かに
 伝える   伝える
```

## 4. 画面・シーン別コピーガイド

### 4-1. オンボーディング ⭐ 強化

初回起動は、店の扉を初めて開ける瞬間。歓迎するが、急がせない。

#### 迎える言葉

```
うちのお店へ、ようこそ。

SNSで見つけたレシピを、今日作れる献立に。
料理の発見と、積み重ねが、ここで始まります。
```

#### 料理目的の選択画面

ラベルは「選んでください」ではなく、「どれに近いですか」の感覚で。

| ❌ 機能的ラベル | ✅ Uchimiseらしいラベル |
|---|---|
| 毎日の食事管理 | 平日の献立を楽にしたい |
| レシピ保存 | SNSのレシピを実際に作りたい |
| 健康管理 | 体に合った料理を続けたい |
| レシピ発見 | 知らない料理と出会いたい |

#### スキップ・後でやる

```
あとで設定する
```

- 「スキップ」は少し冷たい。「あとで設定する」で温度を保つ。

#### 最後の画面（開始前）

```
準備ができたら、扉を開けましょう。

[はじめる]
```

- ボタンラベルは「登録完了」「次へ」ではなく「はじめる」。

---

### 4-2. 空の状態（エンプティステート）⭐ 強化

「何もない」をネガティブに見せない。次の一歩への入口として設計する。

#### コレクション未作成

```
まだ棚が空です。

SNSで見つけたレシピを、ここに並べてみませんか。
```

#### 調理履歴ゼロ

```
調理の記録がまだありません。

作った料理がここに残っていきます。
```

#### 検索結果なし

```
「○○」に合うレシピが見つかりませんでした。

別の言葉で探してみてください。
```

#### 発見タブ（フィード未ロード / 初回）

```
あなたの棚、少しずつ充実させていきましょう。

レシピを保存するほど、あなたらしい棚になっていきます。
```

> `提案が的確になります` はアルゴリズムの説明になるため使わない。「あなたらしい棚になる」という体験の変化として伝える（原則3）。

#### 献立未登録（今日）

時間帯別コピー：

| 時間帯 | ✅ 確定コピー |
|---|---|
| 〜10時 | `今日の朝ごはん、いかがですか。` |
| 10〜14時 | `昼ごはんの支度、はじめませんか。` |
| 14〜21時 | `今夜の一品、考えてみませんか。` |
| 21時〜 | `明日の夕食、決めておきませんか。` |

---

### 4-3. 店主の提案

#### 食材ベースの提案

```
冷蔵庫の鶏肉と大葉で、今夜はどうですか。
今週、タイカレーを保存されていましたね。今日はいかがですか。
```

#### 発見ベースの提案

```
最近、エスニック料理を保存されていますね。今日はこれも、いかがですか。
```

#### 言い回しのバリエーション

| シーン | 使える表現 |
|---|---|
| 食材提案 | `〜で、今夜はどうですか。` |
| 旬の素材 | `今が旬の〜を試してみませんか。` |
| 保存済みレシピ | `〜を保存されていましたね。今日はいかがですか。` |
| 新しい発見 | `〜はいかがですか。材料もそろいそうです。` |

---

### 4-4. 保存・完了系

#### レシピ保存完了

```
レシピを棚においておきました。
```

#### コレクションへの追加

```
「○○」に加えました。
```

#### Phase 2 処理中（PhaseBanner）

```
材料と手順を整理しています…
```

#### Phase 2 完了通知

```
材料と手順が整理されました。
```

#### 調理完了

```
できあがりました。

大葉の香りが立つ一品。
お疲れさまでした。
```

> `できあがりました。` の後に、料理の特徴を一言添える。「お疲れさまでした。」で静かに締める。感嘆符なし。

---

### 4-5. エラー・失敗系

エラーは責めない。何が起きたかを静かに伝え、次の手を示す。

#### 構造：何が起きた ＋ 次にできること

| シーン | コピー |
|---|---|
| 保存失敗 | `うまく保存できませんでした。もう一度試してみてください。` |
| SNS抽出失敗（Private投稿） | `この投稿は非公開のようです。URLのみ保存しておきました。` |
| SNS抽出失敗（字幕なし） | `テキスト情報が取得できませんでした。手動で入力することもできます。` |
| 重複URL | `このレシピはすでに保存されています。` |
| ネットワークエラー | `接続を確認して、もう一度試してみてください。` |

#### やってはいけないエラーコピー

- ❌ `エラーが発生しました。（Error code: 500）`
- ❌ `失敗しました！もう一度試してください！`
- ❌ `申し訳ありません、エラーが発生しました。`

---

### 4-6. 通知・プッシュ

義務感を生まない通知だけを送る。来たいときに来てもらう。

#### 送っていい通知

| タイプ | コピー例 |
|---|---|
| Phase 2完了 | `「チーズタッカルビ」の材料と手順が整理されました。` |
| 週次提案（任意・オプトイン） | `今週の献立、そろそろ考えてみませんか。` |

#### 送ってはいけない通知

- ❌ 連続記録（ストリーク）系：`3日連続で料理しています！`
- ❌ 毎日のログイン促進：`今日のあなたへのおすすめ料理はこちら！`
- ❌ 離脱防止系：`最近来ていませんね。久しぶりに料理してみませんか？`

---

### 4-7. ボタン・CTA ラベル

ボタンラベルは**動詞1〜2語**。何が起きるかを明確に。

| 場面 | ✅ 使うラベル | ❌ 使わないラベル |
|---|---|---|
| 調理を開始 | `調理する` | `スタート` `OK` `はい` |
| 調理モード開始 | `はじめる` | `調理開始` `スタート` |
| 記録の保存 | `記録する` | `完了` `保存` |
| 調理完了後の終了 | `終わる` | `閉じる` `キャンセル` |
| レシピ保存 | `棚に保存する` | `保存` `追加` |
| 献立に追加 | `献立に入れる` | `追加する` `セット` |
| SNS動画を見る | `動画で見る` | `元の動画` `リンク` |
| オンボーディング完了 | `はじめる` | `登録完了` `次へ` |

---

### 4-8. プレースホルダー

プレースホルダーは「入力してください」ではなく、発見のきっかけになる言葉。

| フィールド | ✅ プレースホルダー |
|---|---|
| 検索バー | `レシピ・食材・気分で探す` |
| コレクション名 | `例：平日の夜ごはん` |
| メモ欄 | `次に作るときのメモ` |

---

### 4-9. 削除・破壊的操作の確認ダイアログ ⭐ 追加

何が消えるかを明示し、影響があれば一言添える。ボタンは「キャンセル」でなく**何をするかを表す動詞**にする。

#### 構造：何が消えるか ＋ 影響（ある場合のみ） ＋ 動詞ボタン

#### コレクション削除

```
コレクション「○○」を削除しますか？
保存されているレシピは残ります。

[削除する]　[そのままにする]
```

#### レシピを棚から取り出す

```
このレシピを棚から取り出しますか？

[取り出す]　[そのままにする]
```

#### 調理記録の削除

```
この記録を消しますか？
一度消すと、もとに戻せません。

[記録を消す]　[そのままにする]
```

#### ボタンラベルのルール

| ✅ 使うラベル | ❌ 使わないラベル |
|---|---|
| `削除する` `取り出す` `記録を消す` | `OK` `はい` |
| `そのままにする` | `キャンセル` `いいえ` |

> 「そのままにする」は「キャンセル」より温度が高く、ユーザーに選択を委ねる表現として一貫性がある。

## 5. 絶対禁止リスト（Anti-patterns）

開発中・レビュー時にこれが含まれていたら必ず修正する。

| 禁止パターン | 理由 |
|---|---|
| 感嘆符（！）の使用 | 押しつけがましい |
| 「すごい！」「やったね！」などの過剰な賞賛 | 店主らしくない |
| 「AIが〜しました」「AIが〜を分析」 | 機能説明。体験を壊す |
| 「提案が的確になります」などのアルゴリズム説明 | 原則3違反。体験の変化として言い換える |
| 連続記録・ストリークの言及 | 義務感を生む |
| ログイン促進・離脱防止系の通知 | 義務感を生む。4-6参照 |
| 「おすすめ」単体の多用 | 曖昧。何がおすすめか伝わらない |
| 命令形（〜してください） | 丁寧語の原則に反する |
| `OK` `はい` `いいえ` のみのボタン | 何が起きるか不明 |
| `エラーが発生しました` 単独 | 次の手が示されていない |
| `キャンセル` `閉じる`（確認ダイアログ） | 何をしないのかが不明。4-9参照 |

## 6. コピーチェックリスト

コピーをレビューするときの確認事項。

- [ ] 感嘆符（！）が含まれていない
- [ ] 「AI」「自動」「生成」「的確になります」という言葉が表に出ていない
- [ ] ボタンラベルは動詞で始まっている
- [ ] エラーには「次の手」が含まれている
- [ ] 空の状態には「次の一歩」が含まれている
- [ ] トーンが「丁寧語ベース」になっている
- [ ] 「すごい」「やったね」などの過剰な賞賛がない
- [ ] 連続記録・ストリーク・ログイン促進がない
- [ ] 削除・破壊的操作の確認ダイアログのキャンセル側が「そのままにする」など動詞ラベルになっている
